# FIRST OPERATION DATA INPUT GUIDE

## 1. 목적

- 실제 운영 데이터를 처음 입력하기 전에 필요한 값과 순서를 정리한 가이드다.
- 이 문서는 실데이터를 임의로 채우기 위한 문서가 아니라, 사용자가 실제 값을 준비한 뒤 그대로 입력하기 위한 체크 문서다.

## 2. 입력 전 백업 점검

1. 현재 DB 백업이 있는지 확인한다.
2. 이번 작업에서 생성한 백업 파일을 보존한다.
3. 추가 입력 직전에도 필요하면 한 번 더 백업한다.

현재 생성된 백업:

```text
storage/backups/stock_analyze_before_first_operation_20260703_103226.db
```

## 3. 필요 데이터

실제 입력 전에 아래 값을 먼저 준비한다.

1. 자금 풀 이름
2. 초기 입금액 또는 현재 기준 초기 현금
3. 보유 종목 코드
4. 보유 종목명
5. 보유 수량
6. 평균단가
7. 매수일 또는 기준일

## 4. 자금 풀 생성 기준

- 자금 흐름을 구분해야 하면 계좌 또는 전략 단위로 자금 풀을 분리한다.
- 단일 계좌만 관리할 경우 한 개의 fund pool부터 시작한다.
- 이름은 실제로 구분 가능한 값으로 정한다.

예:

```text
국내주식 주계좌
```

## 5. 초기 입금 입력 기준

- 실제 현재 운용 중인 현금 기준으로 입력한다.
- 과거 입출금 전부를 재현하지 않을 경우, 시작 기준일 시점의 현금 잔고를 초기 입금으로 잡는다.
- 임의 금액을 넣지 않는다.

## 6. 보유 종목 입력 방식

- 종목별 평균단가와 수량을 맞추는 것이 우선이다.
- 과거 모든 체결 내역을 다 입력할지, 현재 잔고 기준으로 단순화할지 먼저 결정한다.
- 단순 시작 방식을 택하면:
  1. 기준 fund pool 생성
  2. 초기 현금 입력
  3. 보유 종목별 매수 거래를 평균단가와 수량 기준으로 입력

## 7. 평균단가 입력 기준

- 현재 보유 수량에 대응하는 실제 평균단가를 사용한다.
- 세금과 수수료를 별도로 포함할지 여부는 일관되게 유지한다.
- 한 종목 내에서 기준이 섞이지 않게 한다.

## 8. 과거 거래 입력 범위 결정 기준

두 방식 중 하나를 먼저 정한다.

1. 전체 이력 방식
   - 과거 매수/매도 거래를 가능한 범위까지 모두 입력
   - 장점: realized/unrealized 흐름이 더 정확함
   - 단점: 입력 시간이 오래 걸림
2. 기준일 시작 방식
   - 특정 기준일 시점 잔고와 보유 종목만 맞춰서 시작
   - 장점: 빠르게 운영 시작 가능
   - 단점: 과거 전체 거래 이력은 남지 않음

## 9. 입력 후 확인할 화면

입력 후 아래 화면을 확인한다.

1. `/portfolio`
   - 자금 풀 표시
   - 현금 잔고 표시
   - 보유 종목 수량/평단 표시
2. `/trades`
   - 입력한 거래 내역 표시
3. `/dashboard`
   - KPI 반영
   - 최근 거래 반영

## 10. 입력 후 확인할 API

입력 후 아래 API를 확인한다.

```text
/api/funds/summary
/api/holdings/summary
/api/portfolio/summary
/api/dashboard/summary
/api/trades
```

핵심 확인 포인트:

```text
funds.active_pool_count >= 1
funds.total_cash 정상
holdings.holding_count 정상
holding quantity 정상
average_price 정상
portfolio.total_asset_value 정상
portfolio.total_cash + holdings market_value 합산 정상
dashboard summary 반영 정상
```

## 11. 주의사항

1. 사용자 실데이터가 준비되기 전에는 입력하지 않는다.
2. 금액, 종목, 수량, 평균단가를 추정해서 넣지 않는다.
3. 첫 실입력 전후로 백업을 유지한다.
4. 실제 Gmail 발송은 이 단계에서 하지 않는다.
5. alerts 화면에서 알림을 먼저 만들 필요는 없다.
6. 입력 후 값이 어긋나면 추가 입력 전에 요약 API부터 먼저 확인한다.
